# Architecture

Hashcrack.ing is a full-stack web application with a Go backend (Fiber), Next.js frontend, PostgreSQL database, and Backblaze B2 cloud storage.

---

## System Overview

```text
+----------------------------------------------------------+
|                    Hashcrack.ing                          |
|                                                          |
|  +-------------------+    +---------------------------+  |
|  |    Next.js 15     |    |     Go Backend (Fiber)    |  |
|  |    Frontend       |◄──►|                           |  |
|  |  React 19         |    |  Routes, Auth, Processing |  |
|  |  Tailwind + Radix |    |  Workers, Hash ID, Tools  |  |
|  +-------------------+    +---------------------------+  |
|                                  |           |           |
|                          +-------+    +------+------+    |
|                          |            |             |    |
|                    +-----v-----+ +---v---+  +------v-+  |
|                    | PostgreSQL | | B2    |  | Discord |  |
|                    | (SQLC)     | | Cloud |  | Bot     |  |
|                    +------------+ +-------+  +--------+  |
+----------------------------------------------------------+
```

## Backend

### HTTP Server (Fiber)

| Setting | Value |
| --- | --- |
| Framework | Fiber v2 |
| Body limit | 1 GB |
| Read/Write timeout | 30 minutes |
| Port (development) | 3001 |
| Port (production) | 3333 |
| CORS | Configured for frontend origin |

### Route Groups

| Group | Authentication | Description |
| --- | --- | --- |
| `/api/register/` | None | Registration with invite token |
| `/api/login/` | None | Authentication |
| `/api/v1/settings/` | Required | User settings and security |
| `/api/v1/admin/` | Required (admin) | User management, invitations |
| `/api/v1/search/` | Required | Hash lookup operations |
| `/api/v1/submit/` | Required | Cracked hash submissions |
| `/api/v1/hashlist/` | Required | Hash list management |
| `/api/v1/wordlist/` | Required | Wordlist management |
| `/api/v1/rules/` | Required | Rule file management |
| `/api/v1/tools/` | Required | Hash ID, secretsdump parser |
| `/api/v1/profile/` | Required | User profile and leaderboard |
| `/api/hooks/discord/` | Webhook key | Discord bot webhooks |

### Background Workers

Post-processing runs asynchronously with a worker pool:

| Setting | Value |
| --- | --- |
| Worker goroutines | 5 |
| Batch insertion workers | 80 concurrent |
| Batch size | 5,000 hashes per batch |
| Chunk types | HASHLIST, RULE, WORDLIST, SUBMITTED_HASHES |

Workers handle:
- Hash list processing (validation, deduplication, digest generation, batch insertion)
- Wordlist processing (validation, storage)
- Rule file processing (validation, storage)
- Cracked hash submission processing (potfile parsing, plaintext association)

## Database (PostgreSQL)

All database operations are generated from SQL queries using SQLC, producing 111 type-safe Go files.

### Core Tables

| Table | Purpose |
| --- | --- |
| `users` | User accounts, roles, API keys, Discord links |
| `sessions` | Authentication sessions with MFA state |
| `pass_keys` | WebAuthn/FIDO2 passkey credentials |
| `mfas` | TOTP MFA secrets |
| `hashes` | Individual hash records with plaintext and Hashcat type |
| `hash_lists` | Hash list collections with metadata |
| `hash_list_hashes` | Many-to-many association (list ↔ hash) |
| `user_submissions` | Track user upload contributions |
| `word_lists` | Wordlist metadata and B2 references |
| `rules` | Hashcat rule file metadata and B2 references |
| `registration_tokens` | Invite tokens with expiration |
| `applications` | User application approval workflow |

## Cloud Storage (Backblaze B2)

All large files are stored in Backblaze B2 with organized bucket paths:

| Bucket Path | Content |
| --- | --- |
| `Wordlists/` | Password wordlists |
| `Rules/` | Hashcat rule files |
| `Hashlists/` | Hash list files |
| `Masks/` | Hashcat mask files |

Features:
- Multi-writer concurrent uploads
- Path-based folder organization
- Direct download links for authenticated users

## Authentication Flow

```text
1. User receives invitation token
   |
   v
2. Register with token + email + password (min 20 chars)
   |
   v
3. Email verification
   |
   v
4. Login → session token created
   |
   v
5. MFA setup required (TOTP)
   |
   v
6. MFA verified → full access granted
   |
   v
7. Optional: register passkey for passwordless auth
```

## Frontend (Next.js)

| --- | --- |
| **Dashboard** | Statistics, recent activity, contribution metrics |
